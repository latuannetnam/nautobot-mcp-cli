## When to Use
- When auditing a Juniper device against its Nautobot CMS records
- When checking for configuration drift across BGP, static routes, interfaces, and firewalls
- When performing pre/post change verification

## Prerequisites
- Device must exist in Nautobot with CMS plugin data populated
- jmcp MCP server connected for live device data collection
- Nautobot API access configured

## Workflow

### Step 1: Get Device Overview from Nautobot
Use `nautobot_device_summary` with device_name to confirm device exists and see interface/IP/VLAN counts.

### Step 2: Collect Live BGP State
Use jmcp `execute_junos_command`:
- `show bgp summary | display json` → extract neighbor list
- Map to: `[{peer_ip, peer_as, local_address, group_name}]`

### Step 3: Compare BGP Against CMS
Use `nautobot_cms_compare_bgp_neighbors`:
- device_name: <device>
- live_neighbors: <JSON from step 2>
- Check report: missing_in_nautobot, extra_in_nautobot (missing_on_device), changed

### Step 4: Collect Live Static Routes
Use jmcp `execute_junos_command`:
- `show route protocol static | display json` → extract route list
- Map to: `[{destination, nexthops: [str], preference, metric, routing_instance}]`

### Step 5: Compare Routes Against CMS
Use `nautobot_cms_compare_static_routes`:
- device_name: <device>
- live_routes: <JSON from step 4>
- Check report: same structure as BGP drift

### Step 6: Review Interface Details
Use `nautobot_cms_get_interface_detail`:
- device_name: <device>
- Review: units, families, filters, policers, VRRP groups, ARP entries

### Step 7: Review Firewall Summary
Use `nautobot_cms_get_device_firewall_summary`:
- device_name: <device>
- Review: filter counts, term counts, policer associations

### Step 8: Interpret Results
- Total drifts = 0 → device is in compliance
- Missing in Nautobot → CMS records need updating (onboard/sync needed)
- Extra in Nautobot → stale CMS records (possible device decommissioning)
- Changed → field-level discrepancies need investigation

## Quick Check (Abridged Workflow)
For a fast drift check without full audit:
1. `nautobot_cms_get_device_bgp_summary` — overview of BGP state in Nautobot
2. `nautobot_cms_get_device_routing_table` — overview of routes in Nautobot
3. Compare with live data using drift tools

## CLI Alternative
```bash
# BGP drift
nautobot-mcp cms drift bgp --device core-rtr-01 --from-file live-bgp.json

# Route drift
nautobot-mcp cms drift routes --device core-rtr-01 --from-file live-routes.json

# BGP summary from Nautobot
nautobot-mcp cms routing bgp-summary --device core-rtr-01

# Interface detail
nautobot-mcp cms interfaces detail --device core-rtr-01

# Firewall summary
nautobot-mcp cms firewalls firewall-summary --device core-rtr-01
```
