## Summary

v1.1 focused on closing the gap between what AI agents needed and what the MCP server provided.
Before v1.1, agents had to rely on config files and manual scripting to perform drift comparisons or even
basic device IP lookups. After v1.1, agents can chain tools directly:

```
jmcp show output → nautobot_get_device_ips → nautobot_compare_device
```
Zero filesystem dependencies. Zero Python scripting.

## Accomplishments

1. **Device-Scoped IP Query** — `nautobot_get_device_ips(device_name)` returns all interface-bound IPs
   using M2M traversal via `ip_address_to_interface` (reliable, avoids Nautobot filter limitations)
2. **Cross-Entity Filters** — `list_vlans(device=...)` and `list_ip_addresses(device=...)` added
3. **Device Summary Tool** — `nautobot_get_device_summary(device_name)` returns interface stats, IP
   counts, VLAN counts, and link-state statistics in one call
4. **Enriched Interface Data** — `nautobot_list_interfaces(include_ips=True)` embeds IPs inline using
   batch M2M query (≤2 API calls, not N+1 per interface)
5. **File-Free Drift Engine** — `compare_device()` with auto-detecting input (flat map or DeviceIPEntry
   list), lenient IP validation, per-interface VLAN drift detection
6. **`nautobot_compare_device` MCP Tool + `verify quick-drift` CLI** — agents and humans can compare
   interface data against Nautobot records without any config file

## Technical Decisions Made

| Decision | Rationale |
|----------|-----------|
| M2M traversal via `ip_address_to_interface` (not direct IP filter) | Nautobot `ip_addresses.filter(device=...)` is unreliable |
| DeviceIPEntry list auto-detection via `isinstance(data, list)` | Allows chaining `nautobot_get_device_ips` output directly |
| Bare IP lenient matching with warnings | Real-world jmcp output rarely has prefix length |
| VLAN comparison only when explicitly provided | Avoids false positives when user only checks IPs |
| Batch M2M query for interface IP enrichment | ≤2 API calls total vs N+1 per interface |
